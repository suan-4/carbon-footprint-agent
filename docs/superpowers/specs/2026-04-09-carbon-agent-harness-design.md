# 碳足迹Agent Harness 设计说明

- 日期：2026-04-09
- 项目：碳足迹Agent
- 主题：第一版 Harness Engineering 设计
- 设计目标优先级：产品可用性 > 系统控制力 > 研发效率
- 第一阶段主战场：知识问答型 + 少量业务助手型

## 1. 设计背景

当前项目已经具备以下基础：

- 微信小程序前端：`miniapp/`
- Express 后端：`server/`
- 轻量知识库：`knowledge/`
- 数据库脚本：`database/`
- 现有 Agent 相关服务：`server/src/services/agentService.js`、`server/src/services/knowledgeService.js`

根据现有文档与项目结构，当前 Agent 已具备基础问答能力，但整体仍偏向“能回答”的原型阶段，尚未形成适合产品场景的稳定执行壳。主要问题包括：

- 不同类型问题可能共用一套回答路径，缺少任务分流
- 上下文装配边界不清晰，容易出现信息污染或无关信息注入
- 输出缺少统一结构与风控校验
- 回答可能难以追溯到具体证据、规则或账户事实
- 缺少围绕失败模式迭代 harness 的最小观测闭环

因此，第一版设计重点不是做多智能体协作，而是为现有 Agent 建立一个稳、可控、可迭代的 harness。

## 2. 设计目标

本设计的目标如下：

1. 让 Agent 在核心产品场景中稳定完成知识解释和低风险业务辅助
2. 显式限制 Agent 权限与行为边界，避免伪执行、越权和过度承诺
3. 通过结构化 trace 和小规模 eval，为后续持续优化提供抓手

明确不在第一阶段追求的目标：

- 不追求多 Agent 协作
- 不追求复杂事务执行编排
- 不追求重型 observability 平台
- 不让 Agent 直接主导数据库写操作

## 3. 方案选择

候选方案包括：

- 方案 A：单 Agent + 路由分流 + 强约束回答
- 方案 B：Planner + Worker 双层 harness
- 方案 C：多专家 Agent 并行协作

本次确定采用方案 A，原因如下：

- 与当前项目阶段最匹配，复杂度最低
- 能快速接入现有知识库、后端服务与业务数据
- 最容易先解决回答稳定性、风险边界和输出一致性问题
- 后续如果业务复杂度提升，可平滑演进为更强的 planner/worker 结构

## 4. 整体架构

第一版 harness 采用五层结构：

### 4.1 入口层

职责：

- 接收用户问题与基础上下文
- 补齐会话所需元数据
- 将请求交给 harness 主编排器

输入建议包含：

- 用户问题
- 用户 ID
- 会话 ID
- 来源页面
- 是否登录
- 时间戳

### 4.2 任务路由层

职责：

- 在回答前先分类请求
- 为后续上下文构建和策略选择提供依据

第一版固定四类路由：

- `knowledge_qa`：规则、FAQ、奖励、隐私、碳知识问答
- `account_explainer`：积分、账户、订单、兑换、打卡结果解释
- `behavior_advisor`：低风险行为建议、任务建议、减碳建议
- `unsafe_or_out_of_scope`：越权、超能力请求、高风险请求、系统不支持请求

路由器除标签外，还应输出：

- `confidence`：高/中/低
- `needs_fallback`：是否应降级为保守回答

### 4.3 上下文构建层

职责：

- 根据路由标签装配不同上下文
- 控制 token 成本与信息污染
- 明确哪些信息允许出现在某类回答中

核心原则：

- 路由决定上下文
- 上下文决定回答质量
- 混合问题先抓主问题，不追求一次答完所有事

### 4.4 回答生成层

职责：

- 在受控上下文内生成最终结构化答案
- 按不同任务类型使用不同 prompt contract

模型的职责应限制为：

- 理解用户表达
- 在给定证据内组织语言
- 做低风险建议的表达
- 对候选路由做语义辅助判断

### 4.5 输出校验层

职责：

- 检查输出结构是否完整
- 校验风险标记与边界约束
- 决定是否放行、降级或拒答

返回前必须完成：

- 结构校验
- 风险校验
- 证据追溯校验
- 回答类型校验

## 5. 核心组件边界

第一版 harness 应坚持：模型负责语义工作，代码负责制度工作。

### 5.1 建议的模块划分

- `agentController`
  - 只负责请求入口与参数整理
  - 不直接拼 prompt
- `agentHarness`
  - 核心编排器
  - 串联路由、上下文、生成、校验、trace
- `intentRouter`
  - 任务分类模块
  - 推荐采用“规则优先 + 模型辅助”
- `contextBuilder`
  - 按路由标签构建上下文
  - 控制信息范围与字段形态
- `policyGuard`
  - 风险边界与权限控制
- `responseValidator`
  - 校验结构化输出与风控要求
- `traceRecorder`
  - 记录整条请求的关键运行信息

### 5.2 模型与代码的职责边界

模型负责：

- 自然语言理解
- 基于证据组织答案
- 低风险建议措辞
- 路由辅助判断

代码负责：

- 路由标签定义
- 数据查询与清洗
- 工具权限控制
- 风险拦截
- 响应结构校验
- trace 记录
- fallback 策略

### 5.3 工具能力分层

第一版以只读能力为主。

允许的只读能力：

- 检索知识库
- 查询积分余额
- 查询兑换记录
- 查询行为打卡记录
- 查询账户摘要

第一阶段禁止 Agent 直接主导的高风险动作：

- 写数据库
- 自动发起兑换
- 自动提交打卡
- 生成正式落库报告

对于这类事务操作，第一版策略为：

- Agent 只负责解释条件、步骤和下一步入口
- 真正执行由前端按钮或明确后端 API 流程完成
- Agent 最多返回结构化 action hint，不直接执行动作

## 6. 路由与上下文构建策略

### 6.1 `knowledge_qa`

适用问题：

- 积分规则解释
- 奖励规则解释
- 隐私政策说明
- 碳知识科普
- 平台 FAQ

允许注入的上下文：

- 用户原问题
- 检索命中的知识块
- 少量系统约束说明

禁止注入的上下文：

- 用户账户明细
- 行为历史
- 无关业务数据

目标：避免把通用规则误说成用户个人事实。

### 6.2 `account_explainer`

适用问题：

- 为什么积分是这个数
- 为什么不能兑换
- 为什么打卡未加分
- 订单或兑换状态解释

允许注入的上下文：

- 用户账户摘要
- 最近相关业务记录
- 对应规则片段
- 必要字段解释

禁止注入的上下文：

- 整库知识全文
- 大段无关 FAQ
- 不相关建议模板

目标：坚持“先给事实，再给规则，再解释关系”。

### 6.3 `behavior_advisor`

适用问题：

- 如何更容易积累积分
- 本周有哪些低门槛减碳行为
- 适合打什么卡
- 如何完成某类环保目标

允许注入的上下文：

- 用户最近行为摘要
- 当前可参与行为清单
- 奖励规则摘要
- 建议输出模板

禁止注入的上下文：

- 敏感账户细节
- 大量隐私条款
- 不相关订单信息

输出风格要求：

- 低风险
- 可执行
- 非命令式
- 2 到 3 条建议为主
- 与现有平台能力强相关

### 6.4 `unsafe_or_out_of_scope`

适用问题：

- 查询他人信息
- 绕过规则
- 伪造打卡或刷积分
- 医疗、法律、财务确定性建议
- 平台不支持的能力请求

允许注入的上下文：

- 用户问题
- 风险规则
- 拒答或降级模板

目标：缩短高风险请求处理路径，减少自由生成空间。

### 6.5 混合问题策略

对于多意图混合问题，第一版不追求完整多任务推理，而是：

- 优先抽取主意图
- 按主意图选择单一路由
- 其余意图通过追加提示引导用户继续追问

例如：

- “为什么我昨天打卡了没加分，顺便告诉我积分规则？”

建议处理为：

- 主路由：`account_explainer`
- 附加引导：如需继续，可进一步解释积分规则

## 7. 输出协议与风控

第一版不让模型直接输出纯自由文本，而是先输出受控结构，再由服务端转换为最终响应。

### 7.1 结构化输出字段建议

- `answer`：面向用户的最终内容
- `answer_type`：`direct_answer` / `explanation` / `suggestion` / `refusal` / `fallback`
- `evidence_refs`：使用到的知识块、规则块或账户事实块
- `risk_flags`：命中的风险标记
- `suggested_actions`：可选，给前端或用户的下一步引导
- `confidence_note`：不确定性提示

### 7.2 硬性风控规则

#### 规则 1：禁止伪执行

如果系统没有真实完成写操作，就不能声称已经完成以下动作：

- 已兑换成功
- 已提交打卡
- 已生成报告
- 已修改账户状态

只能解释：

- 条件是否满足
- 下一步如何操作
- 可以前往哪里完成动作

#### 规则 2：严格区分事实、规则、建议

回答必须清楚区分：

- 系统事实
- 平台规则
- 建议性内容

避免用户把建议误解为系统结论或平台承诺。

#### 规则 3：证据不足时显式降级

以下情况必须触发降级或澄清：

- 检索证据不足
- 账户数据不完整
- 路由低置信
- 问题跨越多个能力边界
- 请求超出现有平台定义

#### 规则 4：高风险问题短路处理

以下请求优先进入拒答或降级模板：

- 越权访问
- 刷积分
- 规避规则
- 医疗、法律、财务确定性建议
- 系统能力外请求

#### 规则 5：回答必须可追溯

后端必须能追溯：

- 路由结果
- 上下文来源
- 证据引用
- 风险标记
- 最终回答类型

## 8. 观测、评估与迭代机制

第一版目标是建立最小可用 eval 闭环，而非构建重型平台。

### 8.1 必须记录的 trace

- `request_trace`
  - 用户问题、用户 ID、会话 ID、来源页面、时间戳
- `routing_trace`
  - 路由标签、置信度、是否降级
- `context_trace`
  - 使用的知识块、账户摘要、建议上下文来源
- `generation_trace`
  - prompt 模板标识、结构化生成结果
- `validation_trace`
  - 命中的风险标记、是否触发 fallback/refusal
- `response_trace`
  - 最终返回内容类型

### 8.2 第一版 eval 集建议

建议先建立小而硬的样本集，每类 10 到 20 条：

- `知识问答集`
  - 测规则解释准确性与引用正确性
- `账户解释集`
  - 测是否能基于给定事实解释账户问题
- `建议质量集`
  - 测建议是否具体、低风险、与平台能力相关
- `风险拦截集`
  - 测越权、伪执行、超能力请求是否被稳妥拦截

### 8.3 eval 判定标准建议

每条样本尽量采用业务可判断标准，而不是抽象评价。建议包括：

- 是否引用了正确规则
- 是否清楚区分事实与建议
- 是否错误声称执行了动作
- 是否在证据不足时保持保守
- 是否触发了正确的风控策略

### 8.4 迭代原则

每次优化 harness 时，优先只改一个主要变量，例如：

- 路由 prompt
- 上下文装配规则
- 风险 guard
- 回答模板

避免同时改动多个关键部件，确保可以定位哪一项修改产生了效果。

## 9. 第一阶段实施边界

第一阶段建议的落地范围：

- 接管现有 Agent 请求入口
- 新增路由、上下文构建、校验与 trace 机制
- 保持只读工具能力
- 优先支撑知识问答与低风险业务解释/建议
- 为未来事务执行预留接口，但不实际放开执行权限

明确暂不纳入第一阶段：

- 多 Agent 架构
- 自动数据库写入
- 复杂跨工具事务编排
- 大规模自动化评测平台
- 向量数据库级别的重型 RAG 升级

## 10. 设计结论

本设计为碳足迹Agent定义了一版适合当前项目阶段的 harness：

- 单 Agent
- 强路由分流
- 按任务装配上下文
- 结构化输出
- 保守风控
- trace + 小规模 eval 驱动优化

它的目标不是追求最复杂的 agent 系统，而是把当前已有的问答原型提升为：

- 更稳定
- 更可控
- 更可解释
- 更容易持续迭代

## 11. 后续衔接

当本设计确认后，下一步应进入 implementation planning，重点拆解：

- harness 主编排器落位
- intentRouter 设计
- contextBuilder 设计
- policyGuard / responseValidator 设计
- trace 与 eval 的最小实现路径
