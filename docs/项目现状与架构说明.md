# 项目现状与架构说明

更新时间：2026-04-20

## 1. 文档目的

这份文档面向继续开发、交接和排查问题的同学，重点说明三件事：

- 当前仓库真实做到哪一步了
- 代码架构是怎么组织的
- 哪些能力已跑通，哪些还没做完

## 2. 当前真实状态

项目当前处于“可运行半成品”阶段，不再是静态原型，也还不是可直接上线的正式版本。

已经跑通的真实链路：

- 微信登录
- 用户会话
- 积分账户
- 行为打卡
- 周报概览
- 商品兑换
- 广场认领与完成认领
- Agent 问答

目前仍属于半成品的原因：

- 没有正式部署域名与线上环境
- 没有后台管理系统
- 没有完成审核上线所需的合规配置
- 广场和商城都还缺少运营级管理能力
- Agent 的 RAG 还是轻量版本

## 3. 系统架构

整体可以分成 5 层：

1. 原型层：`src/html_page/`
2. 小程序前端层：`miniapp/`
3. 接口服务层：`server/`
4. 数据层：`database/`
5. 知识层：`knowledge/`

数据流大致如下：

```text
微信小程序页面
  -> wx.request
  -> Express API
  -> MySQL
  -> 返回业务数据

小碳问答
  -> /api/agent/chat
  -> knowledgeService 检索本地知识
  -> agentService 组装 prompt
  -> OpenAI 兼容模型接口
  -> 返回答案
```

## 4. 前端架构

前端目录在 `miniapp/`，采用微信小程序原生结构。

主要页面：

- `pages/home` 首页
- `pages/log` 日志打卡
- `pages/plaza` 广场互助
- `pages/mall` 积分商城
- `pages/report` 报告
- `pages/profile` 我的
- `pages/redemptions` 我的兑换
- `pages/agent` 小碳问答
- `pages/settings` 接口环境配置
- `pages/privacy` 隐私政策
- `pages/agreement` 用户协议

全局请求封装：

- `miniapp/utils/request.ts`
- `miniapp/utils/request.js`

当前默认 API 地址配置在：

- `miniapp/app.ts`
- `miniapp/app.js`

默认值仍是：

```text
http://localhost:3000/api
```

这意味着：

- 开发者工具本地调试可直接使用
- 真机和上线环境必须替换为正式 HTTPS 域名
- 当前已在“我的 -> 接口环境”提供本地/自定义切换入口

## 5. 后端架构

后端目录在 `server/src/`，按典型 Express 分层组织：

- `config/` 环境配置
- `controllers/` 控制器
- `routes/` 路由
- `services/` 业务逻辑
- `db/` MySQL 连接
- `middlewares/` 中间件
- `scripts/` 辅助脚本

核心服务文件：

- `services/authService.js` 微信登录、会话管理
- `services/appDataService.js` 用户、积分、行为、商品、报告
- `services/plazaService.js` 广场认领与完成认领
- `services/agentService.js` 模型调用
- `services/knowledgeService.js` 本地知识库检索

## 6. 数据层

数据库脚本：

- `database/schema.sql`
- `database/seed.sql`

核心表：

- `users`
- `user_sessions`
- `point_accounts`
- `point_logs`
- `behavior_records`
- `products`
- `orders`
- `carbon_reports`
- `plaza_posts`
- `plaza_claims`

当前已经验证可真实写库的业务：

- 登录后创建/读取用户
- 创建用户 session
- 行为打卡写入 `behavior_records`
- 积分变动写入 `point_logs`
- 商品兑换写入 `orders`
- 广场认领写入 `plaza_claims`
- 广场完成认领后发放积分并关闭需求

## 7. Agent / RAG 架构

当前 AI 模块由两部分组成：

- `server/src/services/agentService.js`
- `server/src/services/knowledgeService.js`

知识来源：

- `knowledge/faq.md`
- `knowledge/rules.md`
- `knowledge/rewards.md`
- `knowledge/privacy.md`
- `knowledge/carbon_basics.md`

当前 RAG 的特点：

- 实现简单
- 维护成本低
- 适合当前阶段快速补知识

当前 RAG 的限制：

- 不是向量检索
- 召回和排序能力有限
- 知识规模变大后会逐渐吃力

## 8. 已完成能力

### 8.1 真实登录

已接入微信登录，走的是：

```text
wx.login -> code -> 后端 code2Session -> openid -> 本地 session token
```

### 8.2 打卡闭环

日志页打卡后会：

- 写行为记录
- 增加积分
- 增加减碳量
- 写积分日志
- 更新报告统计

### 8.3 商城闭环

商城兑换后会：

- 校验库存
- 校验积分余额
- 扣减积分
- 减少库存
- 生成订单
- 写入积分消费日志

并且当前已支持：

- `physical`
- `virtual`
- `badge`

### 8.4 广场闭环

当前广场最重要的 MVP 闭环已经补上：

- 查看待认领需求
- 认领需求
- 生成对接码
- 完成认领
- 发放奖励积分
- 关闭该需求

### 8.5 Agent 问答

问答接口已可用，支持项目内知识检索增强。

## 9. 未完成部分

### 9.1 线上部署未完成

当前仍以本地开发为主，没有正式部署：

- 线上后端服务
- HTTPS 域名
- 小程序合法 request 域名

### 9.2 审核资料未完成

还缺少：

- 隐私政策正式页
- 用户协议正式页
- 小程序服务类目确认
- 审核用功能说明和演示材料

### 9.3 广场仍缺后台能力

当前广场只有用户侧基础闭环，没有：

- 发帖接口
- 审核机制
- 取消认领
- 联系方式管理
- 后台运营页面

### 9.4 商城仍缺履约能力

当前商城只有前台兑换，没有：

- 发货流程
- 核销码/核销后台
- 后台上下架
- 库存管理后台

### 9.5 Agent 仍缺增强

当前 Agent 还缺：

- 向量数据库
- 检索重排
- 引用评分
- 多轮上下文优化

## 10. 推荐继续顺序

如果目标是尽快做成可上线版本，建议按下面顺序推进：

1. 部署后端并拿到 HTTPS 域名
2. 改小程序请求地址并做真机联调
3. 补齐隐私政策、协议与审核资料
4. 补商城核销与广场发帖后台
5. 优化 Agent 的知识库与 RAG
